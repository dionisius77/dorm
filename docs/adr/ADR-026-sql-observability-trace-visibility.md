# ADR-026: SQL Observability & Trace Visibility

- **Status:** Accepted

## Context

This project already treats observability as a first-class concern through ADR-021. SQL execution is one of the most valuable signals for backend developers, QA engineers, DevOps engineers, and SREs, but it also creates a risk of exposing sensitive data if trace payloads are too verbose.

The ORM needs a way to show enough of the generated SQL to explain query behavior, access policy injection, pagination, joins, ordering, and errors while still keeping production deployments safe by default.

## Decision

SQL observability will be configurable through explicit visibility modes.

The ORM will record generated SQL through the query builder path, not raw driver output, so traces reflect ORM behavior rather than backend-specific rewriting.

The default configuration will favor production safety while still leaving room for developer-friendly debugging when explicitly enabled.

## Trace Modes

The ORM will define four SQL trace modes.

### Disabled

No SQL text will be recorded. Only operation metadata and standard tracing fields will be available.

### Metadata

Only metadata will be recorded, such as:

- database system
- operation
- table
- execution duration
- rows affected
- errors

No SQL statement will be captured in this mode.

### Statement

The generated SQL statement will be recorded, but parameter values will not.

This mode is intended to show the query shape while keeping bound values out of the trace payload.

### StatementWithArgs

The generated SQL statement and its parameter values will both be recorded.

This mode is intended primarily for local debugging and should be treated as the most sensitive trace mode.

## Configuration

SQL visibility will be configured through a strongly typed observability configuration rather than booleans.

The API should express intent directly:

```go
orm.Observability{
    TraceSQL: orm.TraceSQLStatement,
}
```

Boolean flags such as `EnableSQLTrace` are rejected because they do not clearly describe the available visibility levels.

## Parameter Masking

When parameters are recorded, masking must be supported.

Common sensitive names should be masked by default, including:

- password
- token
- refresh_token
- access_token
- api_key
- secret

Masking should be configurable so applications can extend the default list without changing core behavior.

The system must never expose sensitive values accidentally.

## Generated SQL

The ORM should trace the SQL generated by the query builder and ORM execution layer.

That visibility is more useful than raw driver output because it allows developers to inspect:

- access policy injection
- automatic filters
- pagination
- joins
- ordering
- query shape

## Access Policy Visibility

SQL traces should make automatic policy injection visible.

If a developer writes a query without explicit filters, the trace should still show the final generated SQL including access policy predicates and soft-delete handling where applicable.

This behavior helps validate access control and reduces the time needed to debug query results that are filtered by policy.

## Observability Integration

SQL tracing must integrate naturally with ADR-021.

Database spans should include:

- SQL visibility mode
- execution duration
- rows affected
- errors
- driver
- dialect

Span names should remain concise and low-cardinality.

## Performance

Tracing should add minimal overhead.

Disabled modes should avoid unnecessary SQL serialization, and the ORM should not pay for expensive formatting work when the trace mode does not require it.

## Security

Production safety is a primary requirement.

The ORM must not expose parameter values unintentionally, and unsafe trace modes must be explicit choices by the application.

## Future Extensibility

The design should leave room for additional observability features without changing the public API.

Examples include:

- SQL formatting
- slow query highlighting
- SQL syntax coloring for UI integrations
- explain plan integration
- query fingerprinting
- sampling
- trace annotations

## Consequences

- Developers can inspect generated SQL with varying levels of detail depending on environment and risk tolerance.
- Access policy injection becomes visible in traces, which improves debugging and security validation.
- Production environments can keep traces safe by default without losing observability entirely.
- Sensitive parameters require explicit handling and masking support.
- The observability layer gains another configuration axis that must remain simple and efficient.

## Alternatives Considered

### Always record full SQL and parameters

### Record only metrics without SQL visibility

### Use a boolean `EnableSQLTrace`

### Depend on raw driver logging only

## Why Alternatives Were Rejected

### Always record full SQL and parameters

Rejected because it creates avoidable security risk in production and exposes sensitive values too easily.

### Record only metrics without SQL visibility

Rejected because metrics alone are not enough to explain ORM behavior, access policy injection, or generated query shape.

### Use a boolean `EnableSQLTrace`

Rejected because a boolean does not capture the meaningful range of visibility levels and leads to ambiguous configuration.

### Depend on raw driver logging only

Rejected because the ORM should expose the SQL it generated, not just whatever the underlying driver happens to report.
